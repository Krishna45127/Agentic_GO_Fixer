# Agentic AI Go Fixer

An agentic AI system that analyzes Go repositories, retrieves relevant source files, generates candidate fixes for GitHub issues, validates the changes, and produces pull request summaries.

The project is designed to be simple, explainable, and easy to evaluate while still demonstrating an end-to-end AI software engineering workflow.

---

# Features

* Repository analysis and indexing
* Lightweight code retrieval
* Issue-aware file selection
* AI-generated fix planning
* AI-generated code modifications
* Patch generation
* Automated validation using:

  * `go vet ./...`
  * `go test ./...`
* Pull request title and description generation
* Sample artifacts for reproducibility

---

# Setup

## 1. Clone This Repository

```bash
git clone <your-repository-url>
cd agentic_go_fixer
```

---

## 2. Create Virtual Environment

### Windows

```bash
python -m venv venv
venv\Scripts\activate
```

### Linux / macOS

```bash
python -m venv venv
source venv/bin/activate
```

---

## 3. Install Dependencies

```bash
pip install -r requirements.txt
```

---

## 4. Install Go

Verify Go installation:

```bash
go version
```

Expected output:

```text
go version go1.xx.x windows/amd64
```

---

## 5. Configure Gemini API

Create a `.env` file:

```env
GOOGLE_API_KEY=YOUR_API_KEY
```

Example `.env.example`:

```env
GOOGLE_API_KEY=YOUR_GEMINI_API_KEY
```

# Workflow

## Step 1: Repository Analysis

The analyzer scans Go files and builds an index.

Output:

```text
repo/index.json
```

---

## Step 2: Retrieval

Relevant files are selected using:

* filename matching
* function/type matching
* content scoring

Output:

```text
output/retrieved_files.json
```

---

## Step 3: Planning

Gemini receives:

* issue description
* selected file
* source code

and generates a fix plan.

Output:

```text
output/fix_plan.md
```

---

## Step 4: Code Generation

Gemini proposes code modifications.

Outputs:

```text
output/generated_patch.diff
output/*.patched.go
```

---

## Step 5: Validation

The system automatically runs:

```bash
go vet ./...
go test ./...
```

Output:

```text
output/test_results.txt
```

---

## Step 6: PR Summary

The system generates:

* PR title
* PR description

Output:

```text
output/pr_summary.md
```

# Design Decisions

The project intentionally avoids:

* Vector databases
* Complex multi-agent architectures
* Heavy orchestration frameworks
* Large autonomous loops

Instead it uses:

* Repository indexing
* Lightweight retrieval
* Gemini-based planning
* Automated repository validation

This keeps the system:

* Simple
* Explainable
* Reproducible
* Easy to evaluate

---

# Limitations

* Generated fixes are candidate solutions and may not exactly match accepted upstream PRs.
* Retrieval quality depends on issue context.
* Complex issues may require additional repository information.
